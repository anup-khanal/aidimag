# Karpathy 3-Layer Integration — Design

> Inspired by Andrej Karpathy's AISN 2026 talk: **Spec → Verifier → Environment**.
> This document maps the framework to aidimag and specifies the next features.
>
> Date: 2026-06-15

---

## Background — The Framework

Karpathy identifies three layers that 10x engineers use when working with AI:

1. **Spec** — deliver your contextual understanding to the AI in a format it can use
   (detailed, scoped, iteratively refined)
2. **Verifier** — evaluation criteria defined upfront, "second AI critic" review, and
   external signal integration so AI can self-check
3. **Environment** — a persistent, improving workspace: `claude.md`, knowledge base,
   custom skills, and explicit guardrails (always / ask-first / never)

The key insight: "You can outsource your thinking, but you can't outsource your
understanding."

---

## How aidimag already maps

| Karpathy Layer | aidimag today |
|---|---|
| **Spec** | Memories injected via MCP `memory_search` / `session_end_extraction`; decisions, conventions, invariants stored as falsifiable claims |
| **Verifier** | `verify/engine.ts`: STATIC_CHECK, TEST_RESULT, EXEC_TRACE, COMMIT_REF evidence; confidence decay; STALE detection; git-hook re-verification |
| **Environment** | `.aidimag/` directory persists across sessions; MCP server auto-injects context; auto-sync keeps team brain fresh |

## What's missing — six features to build

### Feature 1: `dim generate-context` (Environment layer)

**Problem**: Memories only reach agents that support MCP *and* proactively call
`memory_search`. Tools like Cursor (.cursorrules), Copilot (copilot-instructions.md),
and vanilla Claude (CLAUDE.md) inject a static file at session start but have no way
to query aidimag dynamically.

**Solution**: `dim generate-context` auto-generates that static file from your
verified + pinned memories.

```sh
dim generate-context                    # writes CLAUDE.md (default)
dim generate-context --format cursorrules   # writes .cursorrules
dim generate-context --format copilot       # writes .github/copilot-instructions.md
dim generate-context --format all           # writes all three
dim generate-context --watch                # regenerate on every verify/sync
```

**Output structure**:
```markdown
# Project Memory (auto-generated by aidimag — do not edit manually)
# Regenerated: 2026-06-15T10:30:00Z | 12 verified, 2 pinned

## Decisions
- We use LWW sync, not CRDTs — team <10 people, complexity not justified [VERIFIED 📌]
- SQLite + sqlite-vec for zero-dependency local-first store [VERIFIED 📌]

## Conventions
- Always use async/await, never raw .then() chains [VERIFIED]
- Error messages must include the failing input value [VERIFIED]

## Gotchas
- macOS IDE launches from Finder have minimal PATH — use CONSOLE parent env [VERIFIED]

## Architecture
- MCP server → SQLite store → verify engine → CLI/UI/extensions [VERIFIED 📌]

## Invariants
- All DB access goes through src/db/store.ts [VERIFIED]

## Guardrails ⚠️
- 🚫 NEVER: Store API tokens in the memory DB (secrets → OS keychain) [INVARIANT 📌]
- ✅ ALWAYS: Run dim verify before pushing [GUARDRAIL]
- 🤚 ASK FIRST: Before refactoring the auth module [GUARDRAIL]

## Failed Approaches (don't retry these)
- CRDTs for sync — abandoned because operational complexity exceeded team size [VERIFIED 📌]

## Skills (reusable procedures)
- Deploy: 1) dim verify --deep 2) npm run build 3) fly deploy 4) dim sync [SKILL]
```

**Implementation notes**:
- Only includes VERIFIED + UNVERIFIED memories (never STALE/REFUTED)
- Pinned memories always appear; non-pinned sorted by confidence desc
- Header includes generation timestamp + summary counts
- `--watch` mode hooks into the verify/sync/write events
- Git hook integration: optionally regenerate on `post-merge`
- Config: `.aidimag/config.json` → `"generateContext": { "format": "all", "auto": true }`

**Effort**: ~1 day

### Feature 2: GUARDRAIL memory kind (Environment layer)

**Problem**: Karpathy's "bucket tasks into always-do / ask-first / never-do" has no
representation in aidimag today. INVARIANT is the closest, but it lacks the
enforcement-level semantics that AI tools need.

**Solution**: New memory kind `GUARDRAIL` with a required `level` field.

```sh
dim remember "Never commit .env files or secrets to memory payloads" \
  -k GUARDRAIL --level never

dim remember "Always run dim verify before pushing" \
  -k GUARDRAIL --level always

dim remember "Ask before refactoring the auth module" \
  -k GUARDRAIL --level ask-first
```

**Data model addition**:
```typescript
type GuardrailLevel = "always" | "ask-first" | "never";

// In MemoryEntry (only present when kind === "GUARDRAIL")
guardrail_level?: GuardrailLevel;
```

**Behavior**:
- MCP `memory_search` and `memory_get_for_files` include guardrail level in results
- `session_end_extraction` prompt updated: "Respect all GUARDRAIL memories.
  'never' = refuse and explain why. 'always' = do without asking. 'ask-first' =
  ask the user before proceeding."
- `dim generate-context` renders them in a prominent "⚠️ Guardrails" section
- `dim check` (Feature 4) specifically surfaces guardrail violations in diffs
- Dashboard / extensions show guardrails with distinct icons (🚫 / ✅ / 🤚)

**Effort**: ~0.5 day (schema addition + CLI flag + MCP prompt update + context generator)

### Feature 3: `memory_critique` MCP tool (Verifier layer)

**Problem**: Karpathy's "use a second AI as a critic" — but grounded in *real verified
project memory*, not hallucinated opinions. Today, agents can search memories, but
there's no tool designed for "review what I just did against the project's beliefs."

**Solution**: New MCP tool `memory_critique`.

```typescript
// Tool: memory_critique
// Input:
{
  summary: string;        // what the agent just did / plans to do
  files_changed: string[];  // paths touched
}
// Output:
{
  contradictions: Array<{
    memory: MemoryEntry;
    concern: string;   // generated explanation of the contradiction
  }>;
  confirmations: Array<{
    memory: MemoryEntry;
  }>;
  guardrails_violated: Array<{
    memory: MemoryEntry;  // kind=GUARDRAIL
    level: "never" | "ask-first";
  }>;
  missing_coverage: string[];  // areas with no memory → suggest asking user
}
```

**Behavior**:
1. Retrieves all memories scoped to `files_changed`
2. Retrieves all GUARDRAIL memories (project-wide)
3. Compares `summary` against each memory's claim using keyword overlap + semantic
   similarity
4. Returns structured contradictions, confirmations, and guardrail violations
5. Agent can then self-correct before committing

**Integration**: The `session_end_extraction` prompt is updated to suggest calling
`memory_critique` *before* proposing new memories — creating a self-review loop.

**Effort**: ~2 days

### Feature 4: `dim check` pre-commit (Verifier layer)

**Problem**: Verification today runs *after* code lands (post-merge hooks). Karpathy's
"set evaluation criteria upfront" means catching violations *before* they commit.

**Solution**: `dim check` analyzes `git diff --staged` against active memories.

```sh
dim check              # runs against staged changes (for pre-commit hook)
dim check --diff HEAD~1  # check last commit (for CI)
```

**How it works**:
1. Parse the diff → extract changed files + added/removed lines
2. Retrieve all VERIFIED/PINNED memories scoped to those files
3. For each memory:
   - If evidence is `STATIC_CHECK`: re-run it. If it would FAIL after the staged
     changes → **warn**
   - If kind is `GUARDRAIL` with level `never`: pattern-match the diff against the
     claim (e.g., "never import X directly" + diff adds that import → warn)
   - If kind is `CONVENTION` / `INVARIANT`: check if the diff contradicts the claim
     using keyword heuristics (lighter than full verify)
4. Exit 0 (clean) or exit 1 (violations found) + human-readable report

**Git hook**: `dim init` installs an optional `pre-commit` hook:
```sh
# .git/hooks/pre-commit (added by dim init)
dim check --quiet || exit 1
```

**Config**: `"preCommitCheck": true` in `.aidimag/config.json` (opt-in, not default —
avoid annoying devs who haven't bought in yet).

**Effort**: ~3 days

### Feature 5: SKILL memory kind (Environment layer)

**Problem**: Karpathy's "build reusable custom skills" — procedures you do repeatedly
that the AI should know how to execute.

**Solution**: New kind `SKILL` for step-by-step procedures.

```sh
dim remember "Deploy: 1) dim verify --deep 2) npm run build 3) fly deploy 4) dim sync" \
  -k SKILL

dim remember "Onboard teammate: 1) dim cloud init 2) share join code 3) dim sync" \
  -k SKILL
```

**Behavior**:
- Stored like any memory (same verification, confidence, staleness)
- Retrieved by semantic search when the agent's task matches ("how do I deploy?")
- Rendered in `dim generate-context` under a "Skills" section
- Evidence can be attached: e.g., `STATIC_CHECK:fly status` confirms deploy infra exists
- Skills with step markers (1/2/3) get formatted as ordered lists in MCP output

**Effort**: ~0.5 day (just a new kind value + prompt/context formatting)

### Feature 6: Session-start interview prompt (Spec layer)

**Problem**: Karpathy's "have Claude interview you to extract the true goal." Today,
agents jump straight into work without checking what aidimag knows about the area.

**Solution**: Enhanced `session_start` MCP resource/prompt.

**New MCP resource**: `aidimag://session-briefing`
```jsonc
{
  "branch": "feature/AUTH-2100-token-refresh",
  "ticket": { "id": "AUTH-2100", "title": "Token refresh race condition", "type": "bug" },
  "relevant_memories": [
    // top 5 memories scoped to files on this branch's diff
  ],
  "stale_warnings": [
    // any STALE memories in scope — agent should know these are untrusted
  ],
  "guardrails_in_scope": [
    // GUARDRAIL memories that apply to the changed paths
  ],
  "coverage_gaps": [
    // paths with changes but NO memory coverage → suggest asking user
  ],
  "suggested_questions": [
    "The auth module has a STALE convention — has the middleware ordering changed?",
    "No memories cover src/services/tokenRefresh.ts — want me to explore it first?"
  ]
}
```

**Behavior**:
- Auto-injected at session start (MCP resource, like `aidimag://digest` today)
- Includes ticket context if branch matches a ticket pattern
- Identifies gaps → agent asks clarifying questions instead of guessing
- This is "agile speccing" applied to each session

**Effort**: ~1 day

## Why this matters for aidimag's positioning

Today, aidimag is "memory for MCP agents." After these features:

- **`dim generate-context`** makes it "memory for *all* AI tools" — even those without
  MCP support. This 10x's the addressable market overnight.
- **GUARDRAIL + `dim check`** makes it "the CI for AI behavior" — a category that
  doesn't exist yet.
- **`memory_critique`** makes it "the second critic Karpathy says you need" — but
  grounded in verified facts, not another hallucinating model.

The pitch becomes: **"aidimag is the environment layer that makes every AI tool in your
stack 10x more effective — because it feeds them verified context, enforces your
guardrails, and catches contradictions before they ship."**

## Open questions

1. Should `dim generate-context` output be gitignored (regenerated locally) or committed
   (so GitHub Copilot picks it up without local tooling)?
   - Leaning: committed with a `# DO NOT EDIT` header + `.aidimag/config.json` toggle
2. Should `dim check` block commits by default or just warn?
   - Leaning: warn by default, `"preCommitCheck": "block"` opt-in
3. Should GUARDRAIL level be enforced in `memory_critique` (hard error) or advisory?
   - Leaning: `never` = error in `dim check`, advisory in MCP (agent can explain why
     it's overriding, but must state it explicitly)
4. How should `dim generate-context --watch` interact with `dim verify`?
   - Option A: regenerate after every verify run
   - Option B: regenerate only when a memory changes status
   - Leaning: B (less noise)
